一、Relationships
1.Dependencies

A dependency is a semantic relationship between two things in which a change to one thing (the independent thing) may affect the semantics of the other thing (the dependent thing).

  
Choose dependencies when you want to show one thing using another.
2.Generalization
Generalization is a specialization/generalization relationship in which objects of the specialized element (the child) are substitutable for objects of the generalized element (the parent).


3.Associations

An association is a structural relationship that specifies that objects of one thing are connected to objects of another

An association is a structural relationship that describes a set of links; a link is a connection among objects.

there are four adornments that apply to associations
       
You can show a multiplicity of exactly one (1), zero or one (0..1), many (0..*), or one or more (1..*). You can give an integer range (such as 2..5). You can even state an exact number (for example, 3, which is equivalent to 3..3).  
multiplicity *, which is equivalent to 0..*.
Aggregation:  represents a "has-a" relationship.

"whole/part" relationship, in which one class represents a larger thing (the "whole"), which consists of smaller things (the "parts"). 


Hints and Tips

When you model relationships in the UML,

  • Use dependencies only when the relationship you are modeling is not structural.

  • Use generalization only when you have an "is-a-kind-of" relationship; multiple inheritance can often be replaced with aggregation.

  • Keep your generalization relationships generally balanced; inheritance lattices should not be too deep (more than five levels or so should be questioned) nor too wide (instead, look for the possibility of intermediate abstract classes).

  • Use associations primarily where there are structural relationships among objects. Do not use them to show transient relationships such as parameters or local variables of procedures.
